## 目的

Nihongo Pass は、JLPT N2 受験者が試験日までに必要な学習を迷わず続けられるようにする、試験特化型の学習アプリである。

初回リリースでは **JLPT N2 合格** に集中する。ユーザーが現在地を知り、今日やることを進め、弱点を復習し、ミニ模試で合格可能性を更新できる状態を最優先にする。

N5、N4、N3、N1、EJU は将来拡張の導線としてタブを表示する。ただし初回リリースでは、これらのタブを開いた場合は準備中画面のみ表示し、学習コンテンツは提供しない。

## 初回リリースの対象

### 対象ユーザー

- JLPT N2 合格を目指す学習者。
- 試験日までの学習計画を立てたいユーザー。
- 語彙、文法、読解の弱点を短時間で復習したいユーザー。
- 辞書検索、やさしい説明、短いAI補足で理解を進めたいユーザー。
- 日本語だけでは不安があり、英語 UI への切り替えも必要なユーザー。

### 対象範囲

- JLPT N2。
- 語彙、文法、読解。
- ゲスト利用開始。
- ローカル保存を基本とした学習状態管理。
- やさしい日本語と平易な英語の切り替え式 UI。
- 初回起動画面、オンボーディング、10 問診断。
- Home、辞書、Learn、Profile のタブ。
- N2 Learn コンテンツと、N2 以外の準備中タブ。
- 10 分/日の学習計画。
- 約 150 問を初期コンテンツ目標にする。
- Firebase Functions 経由のAI補足生成と共有キャッシュ。

### 初回リリースでは扱わないこと

- N2 以外の実学習コンテンツ。
- EJU 対策コンテンツ。
- 聴解。
- 進学・就職ゴール管理。
- Apple / Google / class code の本番認証接続。
- Firebase ログイン必須化。
- 端末間同期を前提にした体験。
- 本番フル模試。
- Premium 課金導線。
- 管理画面の完全実装。

## 初回起動

最初の画面はログイン画面として見せるが、主導線はゲスト開始にする。

表示するもの:

- アプリロゴ。
- アプリ名。
- 短いメッセージ。
- 小さい補足メッセージ。
- `Login with class code`。
- `Log in with Apple`。
- `Log in with Google`。
- `Get Started`。

`Get Started` を最も強い主アクションにし、オンボーディングへ進める。Apple / Google は表示するが、初回リリースでは本番認証に接続しない。class code はコード入力画面まで表示し、送信時は「無効なコードです」と表示する。将来は学校、先生、クラス教材セットの連携に接続できるようにする。

## コア学習ループ

初回リリースの主要体験は、次の流れに固定する。

```text
初回起動 → オンボーディング → 10問診断 → 学習計画 → 毎日の演習 → 弱点復習 → ミニ模試 → 合格可能性更新
```

各画面は `docs/DESIGN.md` に従い、1 画面につき主要アクションを 1 つに絞る。ユーザーには常に現在位置、進捗、次にやることを表示する。

## 機能方針

### 1. オンボーディング

ユーザーはログインなしで開始できる。

オンボーディングは、画面上部に固定表示する説明とプログレスバー、画面下部に固定表示する `CONTINUE` ボタンを持つ。選択肢は初期状態で選択済みに見せず、ユーザーが明示的に選んだ後だけ選択状態にする。

質問は 12 問前後にする。最低限、次を含める。

- 名前または呼び名。
- UI 言語: `ja` / `en`。
- 母語: 初回リリースでは `ja` / `en` を保存対象にする。
- 何のために日本語を勉強するか。
- 目標レベル: 初回リリースでは `JLPT N2` 固定。
- 現在の日本語レベル。
- 試験予定日。
- 週の学習日数。
- 1 日あたりの学習時間。初期値は 10 分。
- 苦手分野。
- 学習スタイル。
- 通知希望。

プロフィール質問の完了後、10 問診断へ進む。

### 2. 10 問診断

10 問診断は、精密判定ではなく初期学習計画を作るための仮診断として扱う。全員必須で、完了後に Home へ進める。

出題内訳:

- 文字・語彙: 4 問。
- 文法: 3 問。
- 短い読解: 3 問。

聴解は初回リリースでは含めない。診断結果では、仮の現在地、弱点候補、今日から始める学習計画、次の主要アクションを表示する。

10 問診断の直後には合格可能性のパーセント表示を出さない。短い診断結果を過信させないため、合格可能性はミニ模試後から表示する。

### 3. Home

Home は、ユーザーが次に何をすればよいかを最短で理解する画面にする。

固定ヘッダー:

- ユーザーアイコン。
- `Hi, {User Name}`。
- 右寄せの設定アイコン。

コンテンツ:

- 意味や漢字を調べる辞書検索。Home の主役として最上部に置く。
- streak 表示。
- 今日のおすすめ。
- N2 学習カテゴリ。
- 未完了の学習、弱点復習、ミニ模試への導線。

UI は文字を少なくし、アイコン、画像、カード、進捗表示で一目で理解できるようにする。

### 4. 辞書

辞書は、N2 学習中に語彙、漢字、文脈をすぐ確認するための機能にする。

表示するもの:

- 見出し語。
- 読み。
- JLPT レベル。
- やさしい日本語での説明。
- 英語説明。
- 関連するN2タグ。
- 例文。
- 出典表示。

キャッシュ方針:

- 日本語定義と JLPT レベルは端末に同梱する。
- 英語説明、補足、AI生成の言い換えは Firebase Functions 経由で生成し、全ユーザー共通キャッシュに保存する。
- 共有キャッシュは正規化した見出し語、読み、対象言語、生成種別をキーにする。
- ゲストと無料ユーザーにはAI生成の回数上限を置き、既存キャッシュの表示を優先する。

Wikipedia 例文:

- Wikipedia から例文を扱う場合は、短い使用例、記事名、URL、取得日時、ライセンス/出典表示を保存する。
- Wikipedia 本文を問題文として流用しない。
- 例文は語義理解の補助に限定し、公開問題コンテンツとは分けて扱う。

### 5. Learn

Learn は、N2 の出題形式ごとに学習を選べる画面にする。

上部タブ:

- `N5`
- `N4`
- `N3`
- `N2`
- `N1`
- `EJU`

初回リリースで実コンテンツを持つのは `N2` のみ。`N5`、`N4`、`N3`、`N1`、`EJU` を開いた場合は、準備中画面を表示し、N2 学習に戻る導線を置く。

N2 で表示する大問カード:

- 漢字読み。
- 表記。
- 語形成。
- 文脈規定。
- 言い換え類義。
- 用法。
- 文の文法 1。
- 文の文法 2。
- 文章の文法。
- 内容理解 短文。
- 内容理解 中文。
- 統合理解。
- 主張理解 長文。
- 情報検索。

N2 Learn は、[JLPT 公式の試験科目と問題構成](https://www.jlpt.jp/guideline/testsections.html)を参照して、公式問題のコピーではなく、出題形式に合うオリジナル問題を提供する。

### 6. 学習計画

学習計画は、試験日、週の学習日数、1 日あたりの学習時間、10 問診断結果から作る。

初期計画の方針:

- 1 日 10 分で完了できる量にする。
- 新規演習と弱点復習を混ぜる。
- 今日の主アクションを明確にする。
- 完了、未完了、次回復習予定を見えるようにする。

計画はユーザーの演習結果とミニ模試結果で更新する。

### 7. 毎日の演習

毎日の演習は、新規問題と復習問題を混ぜて出す。

演習で保存するもの:

- 問題 ID。
- 回答結果。
- 回答時間。
- 自信度。
- 問題タグ。
- 出題分野。
- 復習候補かどうか。

回答後は、その場で正誤と短い解説を出す。必要に応じてAI補足を表示するが、AI補足は Firebase Functions 経由で生成し、共有キャッシュを優先する。

### 8. 弱点復習

弱点復習に入れる条件:

- 誤答。
- 低自信。
- 同じタグの連続ミス。
- 前回復習から一定期間が経過した問題。

弱点は、N2 の大問タグを中心に管理する。復習画面では、ユーザーが今どの弱点に取り組んでいるかを表示する。復習完了後は、弱点タグの状態と学習計画を更新する。

### 9. ミニ模試

初回リリースの模試は、10〜20 問程度のミニ模試にする。

目的:

- 短時間で本番に近い確認をする。
- 文字・語彙、文法、読解の弱点を更新する。
- 合格可能性を更新する。

本番フル模試、聴解つき模試、厳密な本試験時間再現は初回リリースでは扱わない。

### 10. 合格可能性更新

合格可能性は、ミニ模試後からパーセントで表示する。

初期の算出方針:

- ミニ模試スコア。
- 分野別正答率。
- 弱点タグの残り。
- 学習継続状況。
- 試験日までの日数。

初回リリースでは、機械学習ではなくルールベースの推定でよい。表示時は、数値だけでなく次の改善アクションを必ず併記する。

例:

```text
合格可能性 62%
文法の接続表現を復習すると、次の模試で伸ばしやすいです。
```

## コンテンツ方針

### 問題コンテンツ

初期コンテンツ目標は約 150 問とする。

目安:

- 文字・語彙: 約 60 問。
- 文法: 約 50 問。
- 読解: 約 40 問。

問題には次の情報を持たせる。

- 問題 ID。
- 対象試験。
- 対象レベル。
- 分野。
- 大問タグ。
- 難度。
- 問題文。
- 選択肢。
- 正解。
- 解説。
- AI補足キャッシュキー。
- 出典種別。
- 出典参照。
- 権利メモ。
- 許諾状態。
- レビュー状態。
- 公開状態。

### 外部情報とスクレイピング

スクレイピングは、N2 コンテンツを作るための調査、メタデータ収集、レビュー待ち素材の収集に使える。

公開アプリで利用できるもの:

- 許諾済みコンテンツ。
- 明示ライセンスがあり、条件を満たしているコンテンツ。
- 自社作問。
- AI生成後に人間レビューを通したオリジナル問題。
- 公式情報から抽出した試験構成、分野、出題形式などのメタデータ。

公開アプリで利用しないもの:

- 未許諾の公式過去問。
- 未許諾の公式問題集。
- 第三者の問題文、選択肢、解説、音声スクリプトのコピー。
- 権利状態が不明な本文の流用。

スクレイピングした本文は、公開前レビューと権利確認の対象にする。権利確認が完了していない本文は、公開問題、解説、例文として表示しない。

## データと保存方針

初回リリースはゲスト開始を優先し、ローカル保存を基本にする。Firebase は AI 生成、共有キャッシュ、後続の同期、認証、管理、分析に拡張できるように設計する。

保存対象:

- ユーザープロフィール。
- 試験日。
- 学習時間設定。
- UI 言語。
- オンボーディング回答。
- 10 問診断結果。
- 学習計画。
- 演習履歴。
- 弱点タグ。
- ミニ模試結果。
- 合格可能性履歴。
- 辞書検索履歴。
- AI 共有キャッシュ参照。

実装時は TypeScript 型と Zod スキーマで検証する。React Query は、学習状態、問題セット、演習結果、模試結果、辞書検索、AI 共有キャッシュの取得・更新単位に使う。

## UI と文言

UI は `docs/DESIGN.md` に従う。

文言方針:

- やさしい日本語。
- 平易な英語。
- 1 画面 1 主アクション。
- 進捗を常に見せる。
- 失敗しても責めない。
- 次にやることを短く示す。
- 説明文を増やしすぎず、画像、アイコン、カード、プログレスで理解させる。

言語は併記ではなく切り替え式にする。初回対応は `ja` と `en`。将来的に多言語を増やせるよう、画面文言は翻訳キー化できる構造にする。

## 判断基準

実装中に迷った場合は、次の順に優先する。

1. N2 合格に近づくか。
2. ユーザーが今日やることを迷わないか。
3. 10 分で完了できるか。
4. 弱点復習と模試結果に学習がつながるか。
5. 後で JLPT 他レベルへ横展開しやすいか。
6. コンテンツの権利状態とレビュー状態が明確か。
