# 仕様作成ルール

このフォルダは、Nihongo Pass の機能仕様を実装可能な単位で管理する場所です。
新しい画面、ユーザーフロー、Firebase データ、AI 解説、問題コンテンツ、管理機能を追加する前に、ここへ仕様を作成します。

## 参照する上位ドキュメント

仕様は必ず次のドキュメントに従ってください。

- `docs/DESIGN.md`: UI、文言、画面設計ルール

上位ドキュメントと矛盾する場合は、仕様内に「確認すべき質問」として残し、推測で実装しないでください。

## 配置ルール

新規仕様は機能領域ごとのサブフォルダに置きます。

```text
docs/specs/<area>/<feature-name>.md
```

例:

```text
docs/specs/diagnostic/adaptive-test.md
docs/specs/study-plan/daily-menu.md
docs/specs/mock-exam/mini-mock-result.md
docs/specs/admin/question-review.md
```

命名ルール:

- ファイル名とフォルダ名は英小文字の kebab-case にする。
- 1 ファイルは 1 つの実装可能な機能、画面、またはフローに限定する。
- 複数領域にまたがる大きな仕様は、共通仕様と各機能仕様に分ける。
- 下書きでも、実装に使えない曖昧さは「確認すべき質問」に書く。

## 必須セクション

各仕様には、最低限次のセクションを含めます。

- 目的
- 対象ユーザー
- ユーザーフロー
- 画面/状態
- データ要件
- API/Firebase 要件
- エッジケース
- 実装対象外
- 受け入れ条件
- 確認すべき質問

`docs/specs/_template.md` をコピーして作成してください。

## 仕様の書き方

- ユーザーに見える動作を先に書き、内部実装はその後に書く。
- 画面文言は短く、`DESIGN.md` の「One Screen, One Main Action」に従う。
- Expo Router の画面、React Query の取得単位、Zod の検証対象、Firebase の保存先を必要な範囲で明記する。
- 成功状態、空状態、ローディング、エラー、オフライン、権限不足を明記する。
- 管理画面やコンテンツ作成フローは、公開前レビュー、公開状態、監修者、差し戻しを明記する。
- MVP に入れないものは「実装対象外」に明記し、実装時の判断を残さない。

## コンテンツとスクレイピング

問題、解説、音声スクリプト、選択肢を扱う仕様では、次を必ず守ります。

- スクレイピングは語彙、文法、トピック、難度目安などのメタデータ収集に限定する。
- 公式過去問、公式問題集、第三者の問題文、選択肢、解説、音声スクリプトはコピーしない。
- 公開する問題はオリジナル作問にする。
- 問題データには `sourceKind`、`sourceReference`、`copyrightNote` を持たせる。
- AI 生成を使う場合も、公開前に人間のレビュー状態を持たせる。

## 実装に渡せる状態

仕様は、実装者が追加の判断をしなくても作業できる状態を目指します。

実装可能な仕様の条件:

- 主要ユーザーフローが開始から完了まで書かれている。
- 画面ごとの主アクションが 1 つに絞られている。
- 必要なデータ、保存先、検証ルールが書かれている。
- 失敗時や未設定時の表示が書かれている。
- 受け入れ条件がチェックリストで確認できる。
- 未決定事項が「確認すべき質問」に分離されている。
